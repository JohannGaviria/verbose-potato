## 1. Cloning the repository

```bash
git clone git@github.com:JohannGaviria/silver-enigma.git
cd verbose-potato
```

Copy the environment variables file and fill it in with your local values:

```bash
cp .env.example .env
```

| Variable | Description | Example |
|----------|-------------|----------|
| `APP_NAME` | Application name. | `Verbose Potato` |
| `APP_SUMMARY` | Short summary of the application. | `REST API for library management` |
| `APP_DESCRIPTION` | Detailed description of the application. | `Verbose Potato is a REST API for library management...` |
| `DEBUG` | Enables or disables debug mode. | `True` |
| `ENVIRONMENT` | Application runtime environment. | `development` |
| `BACKEND_PORT` | Port where the API is exposed. | `8000` |
| `BACKEND_WORKERS` | Number of server worker processes. | `4` |
| `CORS_ALLOW_ORIGINS` | Allowed origins for CORS requests. | `http://localhost:8000` |
| `CORS_ALLOW_CREDENTIALS` | Allows credentials to be included in CORS requests. | `True` |
| `POSTGRES_USER` | PostgreSQL username. | `postgres` |
| `POSTGRES_PASSWORD` | PostgreSQL user password. | `password` |
| `POSTGRES_DB` | PostgreSQL database name. | `verbose_potato` |
| `POSTGRES_PORT` | PostgreSQL server port. | `5432` |
| `DATABASE_URL` | Database connection string used by the application (SQLAlchemy Async). | `postgresql+asyncpg://...` |
| `DATABASE_URL_ALEMBIC` | Database connection string used by Alembic for migrations. | `postgresql+psycopg2://...` |
| `DB_POOL_SIZE` | Number of persistent connections maintained in the database connection pool. | `10` |
| `DB_MAX_OVERFLOW` | Maximum number of additional connections allowed when the pool is exhausted. | `5` |
| `DB_POOL_TIMEOUT` | Maximum time (in seconds) to wait for a connection from the pool. | `30` |
| `DB_ECHO` | Enables SQL query logging generated by SQLAlchemy. | `false` |
| `REDIS_URL` | Redis connection URL. | `redis://redis:6379/0` |
| `REDIS_DB` | Redis database index. | `0` |
| `REDIS_HOST` | Redis server host. | `redis` |
| `REDIS_PASSWORD` | Redis server password. | `password` |
| `REDIS_PORT` | Redis server port. | `6379` |
| `REDIS_MAX_CONNECTIONS` | Maximum number of simultaneous Redis connections. | `20` |
| `REDIS_DECODE_RESPONSES` | Whether Redis responses should be automatically decoded into strings. | `true` |

Generate the `JWT_SECRET_KEY` for JWT:

```bash
openssl rand -hex 32
```

Paste the result into `.env`.

> **Never** commit the real `.env` file. It is already excluded via `.gitignore`.

## 2. Dependency management with Poetry

Verbose Potato uses [Poetry](https://python-poetry.org/) as its dependency manager and virtual environment tool.

### Installing Poetry

```bash
curl -sSL https://install.python-poetry.org | python3 -
```

Verify that it's available on your PATH:

```bash
poetry --version
# Poetry (version 2.3.4)
```

### Installing project dependencies

```bash
poetry install
```

Or using the Makefile:

```bash
make setup
```

> **Note:** running this command will also install the pre-commit hooks.
>

This creates an isolated virtual environment and installs all dependencies defined in `pyproject.toml`, including development tools (`ruff`, `mypy`, `pytest`, etc.).

### Useful Poetry commands

```bash
# Add a production dependency
poetry add fastapi

# Add a development-only dependency
poetry add --group dev pytest-asyncio

# Show the dependency tree
poetry show --tree

# Update dependencies while respecting the constraints in pyproject.toml
poetry update
```

## 3. Running the environment

### 3.1 Running the environment without Docker

If you prefer to run the API directly with Poetry and manage the external services (PostgreSQL and Redis) separately:

#### Requirements

- [Python 3.12](https://www.python.org/downloads/release/python-3120/)
- [PostgreSQL](https://www.postgresql.org/docs/)
- [Redis](https://redis.io/docs/latest/)

#### Starting the API

```bash
poetry shell
alembic upgrade head
uvicorn src.main:app --reload --host 0.0.0.0 --port 8000
```

### 3.2 Running the environment with Docker

Docker Compose starts all the necessary services together: the API, PostgreSQL, and Redis.

#### Requirements

- [Docker](https://www.docker.com/) ≥ 24, or Docker Engine + Compose plugin ≥ 2.24
- [Docker Compose](https://docs.docker.com/compose/) ≥ 2.0

#### Starting the full stack

```bash
docker compose up --profile dev --build # or: make up
```

The first run downloads the base images and builds the application image. Subsequent runs are significantly faster if the dependencies haven't changed.

The API will be available at `http://localhost:8000`. Interactive documentation is available at `http://localhost:8000/docs`.

> the default port will be `8000`, but it can be set via environment variables.

### Common commands

```bash
# Run in the background
docker compose --profile dev up -d

# View logs for a specific service
docker compose --profile dev logs -f backend-dev
docker compose --profile dev logs -f postgres

# Rebuild only the application image (e.g. after changing dependencies)
docker compose --profile dev build backend-dev

# Stop all services
docker compose --profile dev down  # or: make down

# Stop all services and remove volumes (deletes the database)
docker compose --profile dev down -v

# Check the status of all services
docker compose ps
```

## 4. PostgreSQL

### Direct connection

```bash
# Locally (requires psql to be installed)
psql postgresql://postgres:password@postgres:5432/verbose_potato

# From inside the container
docker compose --profile dev exec postgres psql -U postgres -d verbose_potato
```
> Use `--profile dev` to connect to the development database or `--profile test` to connect to the testing database.

### Migrations with Alembic

Verbose Potato uses Alembic to version the database schema. All migrations are stored in `alembic/versions/` and are fully reversible.

> The following commands can be run either locally or from the Docker container.

#### Applying migrations

```bash
# Locally
alembic upgrade head

# From the container
docker compose --profile dev exec backend-dev alembic upgrade head
```

#### Reverting the last migration

```bash
# Locally
alembic downgrade -1

# From the container
docker compose --profile dev exec backend-dev alembic downgrade -1
```

#### Reverting to a specific revision

```bash
# Locally
alembic downgrade <revision_id>

# From the container
docker compose --profile dev exec backend-dev alembic downgrade <revision_id>
```

#### Viewing history

```bash
# Locally
alembic history --verbose

# From the container
docker compose --profile dev exec backend-dev alembic history --verbose
```

#### Showing the current revision

```bash
# Locally
alembic current

# From the container
docker compose --profile dev exec backend-dev alembic current
```

### Creating a new migration

Always generate migrations from the SQLAlchemy models instead of writing them by hand.

```bash
# Locally
alembic revision --autogenerate -m "add_notifications_table"

# From the container
docker compose --profile dev exec backend-dev alembic revision --autogenerate -m "add_notifications_table"
```

Review the generated file in `src/shared/infrastructure/database/alembic/versions/` before applying it and make sure the `downgrade()` function is correct.

## 5. Redis

Verbose Potato uses Redis for three distinct purposes:

| Key | TTL | Value | Description |
| --- | --- | --- | --- |
| `cache:books:catalog:{filters_hash}` | 10 min | Serialized list of books | Book catalog according to filters, sorting, and pagination. |
| `cache:loans:catalog:{filters_hash}` | 10 min | Serialized list of loans | Global loan query for `LIBRARIAN`. |
| `cache:loans:member:{member_id}:{filters_hash}` | 10 min | Serialized list of loans | Loan query for a `MEMBER`. |

### Direct connection

```bash
# Locally (requires redis-cli to be installed)
redis-cli -h localhost -p 6379

# From inside the container
docker compose --profile dev exec redis redis-cli
AUTH password
```

### Useful redis-cli commands

```bash
# List all active keys
KEYS *

# Inspect a specific refresh token
GET cache:refresh_token:<token>

# Check the remaining TTL of a key
TTL cache:refresh_token:<token>

# Manually delete a key (useful during development)
DEL cache:refresh_token:<token>

# Show general server information
INFO server

# Clear database 0 (use with caution in production!)
FLUSHDB
```

## 6. Testing

The project has three types of tests, organized into separate directories:

```
tests/
├── unit/          # No DB or Redis. Uses FakeRepositories. Very fast.
├── integration/   # Uses a real PostgreSQL instance. Tests repositories and concurrency.
└── e2e/           # Full HTTP flows against the running application.
```

### 6.1 Testing with Docker (recommended)

Docker Compose uses an isolated database for testing, avoiding any interference with the development database.

```bash
# Run all tests (unit + integration + e2e)
docker compose --profile test run --rm backend-test pytest # or: make test

# Run only integration tests
docker compose --profile test run --rm backend-test pytest tests/integration/ -v # or: make test-integration

# Run only e2e tests
docker compose --profile test run --rm backend-test pytest tests/e2e/ -v # or: make test-e2e

# Run with a coverage report
docker compose --profile test run --rm backend-test pytest --cov=src --cov-report=term-missing # or: make test-coverage

# Run a specific test by name
docker compose --profile test run --rm backend-test pytest -k "test_concurrent_book_confirmation"

# Stop on the first failure
docker compose --profile test run --rm backend-test pytest -x
```

To run integration and e2e tests, the infrastructure services must be running first:

```bash
docker compose --profile test up -d
pytest tests/integration/ -m db
pytest tests/e2e/ -m e2e
```

### 6.2 Testing without Docker (unit tests only)

Unit tests have no external dependencies and can be run directly with Poetry:

```bash
# Run only unit tests
poetry run pytest tests/unit/ -v # or: make test-unit

# Run with a coverage report
poetry run pytest tests/unit/ --cov=src --cov-report=term-missing # or: make test-coverage

# Run in watch mode (requires pytest-watch)
poetry run ptw tests/unit/
```

## 7. Linting and type checking — Ruff + Mypy

### Ruff (linter + formatter)

[Ruff](https://docs.astral.sh/ruff/) replaces flake8, isort, and black with a single, extremely fast binary.

```bash
# Check for linting errors
poetry run ruff check . # or: make lint

# Automatically fix issues Ruff can resolve
poetry run ruff check . --fix # or: make format

# Format the code (equivalent to black)
poetry run ruff format . # or: make format

# Check formatting without modifying files (useful in CI)
poetry run ruff format . --check # or: make check
```

### Mypy (static type checking)

[Mypy](https://mypy.readthedocs.io/) verifies that types are used correctly throughout the codebase.

```bash
# Check types across the whole project
poetry run mypy src/ # or: make check

# Check types of a specific module
poetry run mypy src/modules/Books/

# Generate a type coverage report
poetry run mypy src/ --any-exprs-report .
```

### Running everything together (pre-commit / CI)

```bash
ruff check . && ruff format . --check && mypy src/
```

Or using a single Makefile command:

```bash
# Automatically format and fix code
make format

# Check without modifying files
make check

# Linting and type checking
make lint
```

### Pre-commit hooks (optional but recommended)

```bash
# Install hooks in the local repository
pre-commit install # or: make setup
```

> **Note:** if you already ran `make setup`, the pre-commit hooks were installed automatically.

## 8. Naming conventions

The following rules ensure consistency across the codebase.

### Files

| **Type** | **Convention** | **Example** |
| --- | --- | --- |
| Entity (Domain) | `<aggregate>_entity.py` | `book_entity.py` |
| Value Object (Domain) | `<concept>_vo.py` | `email_vo.py` |
| Repository Port (Domain) | `<aggregate>_repository_port.py` | `book_repository_port.py` |
| Outbound Port (Domain) | `<action>_outbound_port.py` | `notification_outbound_port.py` |
| Domain Service | `<aggregate>_service.py` | `book_service.py` |
| Domain Event | `<action>_<aggregate>_event.py` | `confirm_loan_event.py` |
| Domain Exception | `<aggregate>_exception.py` | `book_exception.py` |
| Use Case (Application) | `<action>_<aggregate>_use_case.py` | `confirm_loan_use_case.py` |
| DTO (Application) | `<action>_<aggregate>_dto.py` | `confirm_loan_dto.py` |
| Mapper (Infrastructure) | `<aggregate>_persistence_mapper.py` | `book_persistence_mapper.py` |
| ORM Model (Infrastructure) | `<aggregate>_model.py` | `book_model.py` |
| Repository Adapter (Infra) | `<orm>_<aggregate>_repository_adapter.py` | `sqlalchemy_book_repository_adapter.py` |
| External Adapter (Infra) | `<tech>_<purpose>_adapter.py` | `smtp_notification_email_adapter.py` |
| Composition (Presentation) | `use_case_composition.py` | `use_case_composition.py` |
| Dependency wiring | `infrastructure_composition.py` | `infrastructure_composition.py` |
| Exception Handlers (API) | `<aggregate>_exception_handlers.py` | `book_exception_handlers.py` |
| Schema | `<action>_<aggregate>_schema.py` | `confirm_loan_schema.py` |
| Router (API) | `<action>_<aggregate>_router.py` | `confirm_loan_router.py` |
| Mapper (Presentation) | `<aggregate>_api_mapper.py` | `book_api_mapper.py` |

### Classes

| **Type** | **Convention** | **Example** |
| --- | --- | --- |
| Entity | `<Aggregate>Entity` | `BookEntity` |
| Value Object | `<Concept>VO` | `EmailVO` |
| Repository Port | `<Aggregate>RepositoryPort` | `BookRepositoryPort` |
| Outbound Port | `<Action>OutboundPort` | `NotificationOutboundPort` |
| Domain Service | `<Aggregate>Service` | `BookService` |
| Domain Event | `<Action><Aggregate>Event` | `ConfirmLoanService` |
| Exception | `<Aggregate>Exception` | `BookException` |
| Use Case | `<Action><Aggregate>UseCase` | `ConfirmLoanUseCase` |
| Command (input DTO) | `<Action><Aggregate>CommandDto` | `ConfirmLoanCommandDto` |
| Response (output DTO) | `<Action><Aggregate>ResponseDto` | `ConfirmLoanResponseDto` |
| ORM Model | `<Aggregate>Model` | `BookModel` |
| Persistence Mapper | `<Aggregate>PersistenceMapper` | `BookPersistenceMapper` |
| Repository Adapter | `<Tech><Aggregate>RepositoryAdapter` | `SQLAlchemyBookRepositoryAdapter` |
| External Adapter | `<Tech><Purpose>Adapter` | `SMTPNotificationEmailAdapter` |
| Request Schema | `<Action><Aggregate>RequestSchema` | `ConfirmLoanRequestSchema` |
| Response Schema | `<Action><Aggregate>ResponseSchema` | `ConfirmLoanResponseSchema` |
| Router | `<Aggregate>Router` | `BookRouter` |
| API Mapper | `<Aggregate>ApiMapper` | `BookApiMapper` |
| Exception Handler | `<Aggregate>ExceptionHandler` | `BookExceptionHandler` |
| Use Case Composition | `<Action><Aggregate>Composition` | `ConfirmLoanComposition` |

### Variables and functions

```python
# Variables and parameters: snake_case
book_id: UUID
member_id: UUID
available_copies: int

# Functions and methods: snake_case, verb first
async def get_book_for_update(book_id: UUID) -> Book: ...
async def reserve_atomic(items: list[BookItemVO]) -> None: ...
async def record_stock_movements(movements: list[StockMovementVO]) -> None: ...

# Module-level constants: SCREAMING_SNAKE_CASE
MAX_RETRY_ATTEMPTS = 3
DEFAULT_PAGE_SIZE = 20

# Enums: PascalCase for the class, SCREAMING_SNAKE_CASE for the values
class BookStatus(str, Enum):
		LIBRARIAN = "LIBRARIAN"
	  MEMBER = "MEMBER"

class StockMovementType(str, Enum):
		ACTIVE = "ACTIVE"
	  RETURNED = "RETURNED"
```

### Additional rules

- Use case methods are always called `execute()`.
- Repository methods use descriptive verbs: `get_by_id()`, `get_book_for_update()`, `save()`, `list_by_member()`.
- Ports (ABCs) contain no logic — only abstract method signatures decorated with `@abstractmethod`.
- Domain exceptions must inherit from the base class `BaseDomainExcepcion(Exception)`.
- Domain events are immutable dataclasses (`@dataclass(frozen=True)`).
